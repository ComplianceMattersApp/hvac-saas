## Action / Route Contract

### Completion Report route

- Route: `/jobs/{jobId}/tests?t=completion_report`
- Optional selected system query: `s={systemId}`
- Builder: `withS("completion_report", selectedSystemId)` in `app/jobs/[id]/tests/page.tsx`
- Report behavior: report-first ECC print/download view; normal test workspace panels are hidden while focused.
- Refrigerant evidence behavior: the focused Completion Report path loads tagged refrigerant charge evidence images and has dedicated tests in `completion-report-refrigerant-evidence-wiring.test.ts`.

### `markCertsCompleteFromForm`

Source: `lib/actions/job-ops-actions.ts`

Form fields observed:

- Required: `job_id`
- Optional/used: `return_to`

Server-side behavior observed:

- Requires authenticated user.
- Requires internal ops access for the scoped job.
- Requires operational job ops entitlement.
- Reads the current closeout job snapshot.
- Blocks failed/retest states with `notice=failed_requires_retest`.
- For ECC, reads ECC runs and blocks completed failed runs unless correction-review resolution exists.
- Blocks if field work is not complete with `notice=field_not_complete`.
- Resolves billing truth before cert closeout.
- Applies ECC permit-needed blocker before marking certs complete.
- Updates `jobs.certs_complete=true`.
- Heals `invoice_complete=true` only when billing truth is already satisfied and the lightweight projection is stale.
- Recomputes `ops_status` through the shared resolver.
- Inserts a `job_events` ops update.
- Revalidates `/jobs/{jobId}`, `/ops`, `/ops/closeout-queue`, and `/reports/closeout`.
- Redirects through `buildJobOpsRedirectPath` using `return_to`, with banners:
  - `certs_closeout_closed`
  - `certs_closeout_saved`
  - `certs_closeout_failed`
  - `permit_needed`

### Related closeout actions

The shared status surface also exposes or routes to closeout-adjacent actions:

- Internal invoice draft creation via `createInternalInvoiceDraftFromForm`.
- Existing invoice workspace link: `/jobs/{jobId}/invoice#invoice-workspace`.
- External billing completion via `completeDataEntryFromForm` or `markInvoiceCompleteFromForm`.
- Permit information/edit through current mobile `mobile-permit-info`, where permit edit currently reuses `updateJobScheduleFromForm` with schedule and permit fields.

These are closeout/billing truth surfaces, not just certification display. They should remain outside this narrow Completion Report parity slice unless separately audited.

## Proposed V2 Placement

| Surface | Proposed V2 placement | Reason |
| --- | --- | --- |
| Completion Report route link | More Details / Tools, under Admin / Records, or Compliance Work for ECC-only jobs | It is a report/workspace, not the primary field action. It should be reachable without crowding the hero or replacing tests. |
| Certs Sent action | Status / Next Step surface only, via existing `MobileJobStatusActionSurface` | Already native and guarded by closeout/certs/permit/failure state. |
| Passive `Certs sent` status | Status / Next Step or report route, not duplicated elsewhere | Avoid redundant certification messaging. |
| Permit/certs blocker details | Compliance Work supporting row plus current/standard permit target | Permit edit is coupled to current mobile schedule/permit form fields. |
| Billing/closeout review | Billing / Closeout card and standard/current anchors | This keeps invoice/payment/external billing truth out of a visual-only promotion slice. |

## Risk Review

- Direct Completion Report route linking is low to medium risk because it uses an existing route/workspace and does not duplicate mutation forms.
- Native cert closeout expansion is high risk because `markCertsCompleteFromForm` updates cert truth, can heal invoice projection, recalculates `ops_status`, inserts events, revalidates closeout reports, and redirects with state banners.
- Permit-related cert blockers are high risk because permit edit is currently part of the current mobile permit panel and reuses schedule action fields.
- Billing/external billing closeout is high risk because billing truth can come from durable invoice state, external billing disposition, and lightweight projections.
- Completion Report itself is ECC/test-workspace-specific; promoting it for Service jobs would be misleading unless a separate Service completion report surface exists.
- V2 should not create a second `Certs Sent` button outside the shared status surface or Completion Report route.

## Recommendation

Recommendation: **Implement direct Completion Report route/link in V2 next, but keep certification and closeout mutations in the existing shared status surface or existing report/workspace routes.**

Safe next slice:

1. Add an ECC-only `Completion Report` row in V2 More Details / Tools or Compliance Work.
2. Link to the existing route `/jobs/{jobId}/tests?t=completion_report`.
3. Do not add a native V2 `Certs Sent` form.
4. Do not change `markCertsCompleteFromForm`.
5. Do not change Completion Report route behavior, report print/download controls, tests truth, cert truth, permit truth, billing truth, or closeout truth.

Do not implement native certification/closeout action expansion next. That should remain a separate closeout/billing/permit audit because the current action contract is intentionally stateful and source-of-truth-heavy.

## Stop-And-Discuss Concerns

- If owner expects a non-ECC Service Completion Report, source did not show a matching current mobile/report route in this audit. That should be clarified before adding Service-facing copy.
- If V2 needs to show selected system-specific Completion Report links, confirm whether V2 has a trustworthy selected system context. Otherwise use the route default and let the tests workspace own system selection.
- If owner wants Certs Sent directly from a V2 report row, stop. That would duplicate an existing mutation outside the shared status surface and should be handled only after a closeout action audit.
